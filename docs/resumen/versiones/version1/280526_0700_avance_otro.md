## 3. Módulo de Materiales — control de activación/desactivación

### Gaps corregidos en vistas
Tres vistas mostraban la pestaña "Materiales" sin comprobar si el módulo estaba activo:

- **`resources/views/livewire/borradores/ver.blade.php`**
- **`resources/views/livewire/proyectos/ver.blade.php`**
- **`resources/views/livewire/proyectos/editar.blade.php`**

En los tres archivos se aplicó el patrón `array_values(array_filter([...]))` con condición `\App\Support\Modulos::materialesAvanzado() ? [...] : false` para la pestaña, y `@if(\App\Support\Modulos::materialesAvanzado())` envolviendo el bloque de contenido.

### API sin protección corregida
- **`app/Http/Controllers/Api/ProyectoOpcionesController.php`**
  - La consulta de materiales ahora devuelve `collect()` vacío si el módulo está desactivado, en lugar de devolver siempre la lista completa

### Botón imprimir en albaranes
- **`resources/views/livewire/albaranes/editar.blade.php`**
  - La opción "Con materiales" del desplegable de impresión queda oculta con `@if(\App\Support\Modulos::materialesAvanzado())`
  - Si el módulo está desactivado, solo aparece "Sin materiales"

## 4. Auditoría y corrección de roles y permisos

### Problema detectado: middlewares demasiado permisivos
Cuatro rutas de creación/edición usaban el permiso `ver` en lugar del permiso correcto:

- **`routes/web.php`**
  - `/clientes/crear` → `can:clientes.ver` **corregido a** `can:clientes.crear`
  - `/clientes/{id}/editar` → `can:clientes.ver` **corregido a** `can:clientes.modificar`
  - `/proyectos/crear` → `can:proyectos.ver` **corregido a** `can:proyectos.crear`
  - `/proyectos/{id}/editar` → `can:proyectos.ver` **corregido a** `can:proyectos.modificar`

### Problema detectado: botones "Nuevo" con `@can` incorrecto
Cuatro vistas mostraban el botón "Nuevo cliente / Nuevo proyecto" a cualquiera con permiso `ver`:

- **`resources/views/livewire/clientes/ver.blade.php`** → `@can('create', Cliente::class)`
- **`resources/views/livewire/clientes/editar.blade.php`** → `@can('create', Cliente::class)`
- **`resources/views/livewire/proyectos/ver.blade.php`** → `@can('create', Proyecto::class)`
- **`resources/views/livewire/proyectos/editar.blade.php`** → `@can('create', Proyecto::class)`

Ahora middleware, Policy y botón de vista comprueban de forma coherente el mismo permiso en las tres capas.

### Estado del sistema de permisos tras la auditoría
- 69 permisos definidos, 4 roles (superadmin, administrador, trabajador, responsable)
- Jerarquía correcta; datos sensibles restringidos a superadmin
- ~30 permisos definidos en el seeder para funcionalidad futura aún no implementada (no son un riesgo de seguridad)
- Policies registradas automáticamente para 12 modelos, todas coherentes
