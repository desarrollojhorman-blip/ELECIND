ELECIND

bellon en el servidor sin depender de nadie para cada cambio. La idea es que puedas subir 
actualizaciones completas cuando tengas una nueva versión del proyecto, pero también 
que puedas entrar y modificar archivos concretos si lo necesitas como desarrollador.

4. Qué puedes tocar y qué debes tratar con cuidado
Puedes modificar los archivos del proyecto porque eres el desarrollador responsable. Aun 
así, hay archivos y carpetas que conviene tratar con cuidado porque afectan directamente a 
producción.
La norma práctica es sencilla: si vas a cambiar código, vistas, rutas, controladores, modelos 
o migraciones, puedes hacerlo. Si vas a tocar 
.env
, 
storage
, permisos, base de datos, 
servidor web o servicios, avísame antes o asegúrate de tener claro lo que estás haciendo.

5. Forma recomendada de subir una versión completa
Cuando tengas una nueva versión lista, lo más limpio es subir un ZIP del proyecto y ejecutar 
el script de despliegue. Ese script hace todo lo necesario para que Laravel quede bien 
actualizado.
El ZIP debe contener la carpeta raíz del proyecto Laravel. Dentro deben estar archivos como:
Plain Text
artisan
composer.json
composer.lock
package.json
package-lock.json
app/
bootstrap/
config/
database/
public/
resources/
routes/
No hace falta incluir 
vendor
 ni 
node_modules
, porque el servidor los genera 
automáticamente.
Tampoco debes incluir el 
.env
 local para no sobrescribir el 
.env
 de producción.

Desde tu ordenador, estando en la carpeta donde tengas el ZIP, ejecuta:
Bash

Si tu archivo tiene otro nombre, cambia

7. Cómo ejecutar el despliegue
Una vez subido el ZIP, ejecuta el despliegue con este comando:
Bash
sudo /usr/local/sbin/deploy-uceda-bellon /home/jhorman/uploads/uceda-bellon-main
Si tu ZIP tiene otro nombre, cambia la ruta. Por ejemplo:
Bash
sudo /usr/local/sbin/deploy-uceda-bellon /home/jhorman/uploads/elecind-nueva-ve
El despliegue irá mostrando pasos como estos:

[3/10] Sincronizando código sin sobrescribir .env ni storage
[4/10] Composer install
[5/10] Frontend build
[6/10] Migraciones y enlaces
[7/10] Cachés Laravel
[8/10] Permisos para desarrollador y proceso web
[9/10] Reinicio servicios app
[10/10] Limpieza temporal
OK despliegue completado
Si termina con 
OK despliegue completado
, la subida ha terminado correctamente.
8. Qué hace el script de despliegue

El script no se limita a copiar archivos. Hace una actualización completa de Laravel para 
evitar errores típicos de producción.
Esto evita tener que ejecutar muchos comandos manuales cada vez.

12. Qué hacer si la web da error 500
Si aparece un error 500, lo primero es mirar el log de Laravel:
Bash

cd /opt/entreredes/apps/uceda-bellon/current
 tail -n 100 storage/logs/laravel.log
Si el error es por caché, prueba:
Bash
php artisan optimize:clear
php artisan config:cache
php artisan route:cache
php artisan view:cache
Si el error aparece justo después de subir una versión, revisa especialmente:
No repitas despliegues muchas veces sin revisar el error. Si falla, copia el error exacto y me 
lo pasas.
13. Qué hacer si falla 
npm run build
Si el despliegue falla en 
npm run build
, normalmente el problema está en JavaScript, CSS, 
Tailwind, Vite o imports del frontend.
Antes de subir una versión, si has tocado frontend, conviene probar en local:
Bash
npm install
npm run build
Si en local falla, también fallará en el servidor.

composer install
Si falla 
composer install
, suele haber un problema con 
composer.json
, 
composer.lock
 o 
alguna extensión PHP necesaria.
Antes de subir una versión que cambie dependencias PHP, prueba en local:
Bash
composer install
Si has añadido paquetes, asegúrate de subir también 
composer.lock
 actualizado.
15. Qué hacer si hay migraciones nuevas
Si añades migraciones en 
database/migrations
, el script de despliegue las ejecutará 
automáticamente con:
Bash
php artisan migrate --force
Antes de subir una migración, pruébala en local o en entorno de pruebas. Una migración 
mal hecha puede modificar datos de producción.
Si tienes dudas con una migración destructiva, avísame antes de subirla.
16. Backups antes de cada despliegue
El script crea un backup antes de sincronizar la nueva versión. Los backups se guardan en:
Plain Text

Puedes verlos con:
Bash
ls -lh /opt/entreredes/apps/uceda-bellon/backups
No borres backups sin confirmarlo antes.

19. Cosas que no quiero que hagas sin avisarme
Puedes trabajar sobre el proyecto, pero hay ciertas acciones que prefiero que no hagas sin 
avisar antes porque pueden afectar al servidor completo o a datos de producción.
La idea es que tengas libertad para desarrollar y subir cambios, pero sin tocar partes del 
servidor que afectan a otras aplicaciones o servicios.
