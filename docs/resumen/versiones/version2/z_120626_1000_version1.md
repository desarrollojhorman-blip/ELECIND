## 1 · Stack y forma del producto

- **Backend**: Laravel 12 + PHP 8.3.
- **Frontend web**: Livewire 3 + Alpine + Blade + TailwindCSS.
- **Frontend móvil**: Livewire (`/m/...`) + layouts dedicados.
- **BD**: MySQL (XAMPP local · LucusHost/servidor en producción).
- **Permisos**: `spatie/laravel-permission`. **Audit**: `spatie/laravel-activitylog`. **Media**: `spatie/laravel-medialibrary`. **PDF**: `carlos-meneses/laravel-mpdf`. **Excel**: `maatwebsite/excel`. **Fechas**: Flatpickr. **Firma**: Canvas API + Alpine.
- **Single-tenant** (personalizado para Elecind, sin multi-tenant).
- **4 roles del sistema** + 1 personalizable: Superadmin (web+móvil), Administrador (web), Jefe de equipo (web, scoped por cliente), Trabajador (móvil), Responsable (móvil/token email).

## 2 · Qué se entrega en la v1 (módulos operativos)

### 2.1 Base (Fase 0)
- Laravel + estructura, layouts `web` y `mobile`, login por username, middleware web/móvil, CI básico, Pint/Larastan, login con `Recuérdame` (cookie 5 años) + `autocomplete="username"`/`"current-password"` para que el navegador ofrezca guardar.

### 2.2 Catálogos base (Fase 1)
CRUDs completos con búsqueda, filtros, papelera (soft delete + restore), exportar Excel/PDF y/o importar Excel:
- **Clientes** (codigo_cliente correlativo, papelera con `clientes.gestionar_papelera` superadmin-only, import/export)
- **Proyectos** (con tipos al vuelo, `codigo` + `codigo_secuencial`, estados `activo/inactivo/cerrado`)
- **Grupos de proyecto**
- **Tipos de proyecto** (gestionables desde el form de proyecto)
- **Materiales** + **Familias de material** + **Números de pedido** (módulo "avanzado" activable; ver §2.7)
- **Conceptos** (catálogo global + pivot N:M con proyectos, papelera, import/export)
- **Usuarios** (jerarquía por nivel, scoping por rol, `id` como código visible, número empleado HR, papelera, import/export, **tarifas €/h** ahora visibles para administrador)

### 2.3 Empresa y Ajustes
- **`/configuracion/empresa`**: nombre, razón social, CIF, dirección, **móvil**, **web**, email contacto, email notificaciones, **logo principal** + **logo albarán** + **logo app** (cada uno con cascada, detección automática de aspect ratio, zoom configurable 80-130%).
- **`/configuracion/ajustes`**: plantilla numeración albarán, prefijo proyecto, límites archivos, colores app, toggle del módulo materiales avanzado, **SMTP** (host/puerto/cifrado/usuario/contraseña/from), **Probar conexión**.
  - SMTP password **write-only** (no viaja al navegador, cifrada en BD, botón "ojo" eliminado).
- **`/configuracion/api`** (solo superadmin): CRUD de tokens API externos.
- **`/configuracion/logs`** (visor de Activity Log) con filtros estilo Horas (Usuario searchable-select, Entidad, Evento, Buscar, Desde/Hasta, Limpiar filtros).
- **`/configuracion/licencias`** (info de licencia + producto + módulos). Producto: `versión 3.26` actual. Sección MÓDULOS ocultada en blade.
- **`/configuracion/roles`**: CRUD de roles personalizados + edición de permisos (sistema protegido, niveles, delegación, ámbitos).

### 2.4 Albaranes core (Fase 2)
- **5 tablas**: `albaranes`, `albaran_lineas_personal`, `albaran_lineas_material`, `albaran_firmas` (refactor polimórfico → `firmas`), `albaran_tokens_firma` (→ `token_firmas`).
- **3 estados** (reducidos de 5): `pendiente_firma → firmado → facturado` (eliminados `borrador` y `archivado` con migración + remapeo).
- **`tipo_hora`** en cabecera con 4 valores (laboral / lab.noche / festivo / fest.noche). Líneas personal con `horas` + `horas_extra`.
- **Móvil**: pantalla "Nuevo parte" + "Ver parte" con cascada de selects, líneas de personal y material.
- **Firmas**:
  - Canvas para firma in-situ.
  - Token email + ruta pública `/firmar/{token}` sin auth (rediseñada con scroll horizontal en móvil).
  - Doble firma (trabajador + responsable). PDF se regenera mostrando firmas.
  - Disco de firmas migrado a `public` (las del móvil ahora se ven en web y PDF).
  - Select firmante en web muestra trabajadores de las **líneas**, no del proyecto.
  - Móvil acepta firma a usuarios con `firma_trabajador_user_id ≠ creado_por` (no más 403).
- **PDF de albarán** (mPDF) **rediseñado**: monocromo (negro/grises), logo más grande, cabecera con tlf+móvil+web+email, bloque "DATOS DEL CLIENTE" titulado fuera del rectángulo, tablas con borde+divisores verticales, sin marco exterior, firmas sin caja, una sola columna **"Horas"** (total = `horas + horas_extra`, sin desglose para no exponer extras al cliente).
- **Correo solicitud-firma** (Mailable + blade) reescrito **100% en tablas anidadas** (bulletproof para Gmail/Outlook), espejo del PDF, preheader único por envío (evita el "trim" de Gmail cuando se envía a uno mismo), sin sección de materiales (decisión: en el correo no van).
- **Pantalla pública de firma** rediseñada estilo PDF, con scroll horizontal en móvil (`overflow-x-auto` + `min-w-[680px]`), sin materiales.
- **Snapshots** en líneas (Observers) congelan datos al firmar (regla `isDirty`).
- **Adjuntos** vía medialibrary (límites configurables en Ajustes).

### 2.5 Borradores (Fase 3)
- **Parte personalizado móvil** crea **Borrador** con número `BOR-####` (no albarán). Cascada cliente → proyecto → concepto → responsable con opción **"Otro"** (texto libre) en cada nivel.
- **CRUD web** de borradores (Index con paginación, filtros, papelera, colores en iconos de acción, **scoping** para jefe de equipo).
- **Form web** ahora con toggle "Seleccionar existente / Texto libre" en **cliente, proyecto, concepto y responsable** (`responsable_texto` recién añadido al form).
- **Asistente de conversión** `/borradores/{id}/convertir` (página propia, 5 pasos, todo en una transacción al confirmar):
  1. Cliente (elegir existente o crear nuevo).
  2. Proyecto (elegir del cliente o crear nuevo bajo ese cliente).
  3. Concepto (elegir del proyecto o de otros, o crear).
  4. Responsable opcional (sin / elegir del cliente / crear con usuario+contraseña+nombre+email opcional).
  5. Confirmar (resumen + commit transaccional → albarán `pendiente_firma`).
- Botón "Convertir a albarán" en Ver y en Editar redirige al asistente.

### 2.6 Ausencias e Incidencias (Fase 4)
- **Ausencias web** (`/ausencias`): listado + CRUD + papelera + filtros + chips + 4 estados + import/export Excel/PDF. Permiso `ausencias.aprobar`.
- **Ausencias móvil**: solicitar ausencia, ver propias, banner según estado, rediseño de tarjeta (papelera lateral solo si pendiente).
- **Incidencias web** (`/incidencias`): listado + gestión de estado/resolución + filtros (incluye prioridad). Modal de gestión separa bloque readonly + sección editable (estado + resolución).
- **Incidencias móvil**: crear desde menú ⋮, listado con búsqueda sticky, prioridad como select.
- Enum `TipoIncidencia` reducido a 3 (`albaran`/`ausencia`/`otro`) con migración remapeadora.

### 2.7 Módulo "Materiales avanzado" activable
- Flag `empresa.modulo_materiales_avanzado`.
- Cuando OFF: sidebar oculta "Pedidos", editor de Material simplificado, opción "Con materiales" del PDF de albarán oculta, API del proyecto no devuelve materiales, pestañas de borrador/proyectos no aparecen.
- Cuando ON: todo el módulo de pedidos/familias/precios visible.

### 2.8 Control de Horas + Exportación
- **`/horas`**: detalle diario por empleado, filtros (Trabajador, Cliente, Proyecto, Estado albarán, Desde, Hasta), filtros con códigos delante (Trabajador: `001 · Apellido Nombre`; Cliente: `COD · Nombre`; Proyecto: `(COD) Nombre`), pills "Mis clientes" / "Todos" / "Por revisar" para usuarios scoped, **paginación** (selector 5/10/25/50/100/250/500), barra de totales 2×4 con `SUM` agregado en SQL (no en PHP).
- **Exportar Excel** (`maatwebsite/excel`) con 19 columnas incluyendo tasas y importes calculados (laboral/festivo/extras con sus tasas correspondientes). Solo internamente — sigue mostrando desglose normales/extras.
- **Móvil** `/m/resumen`: el operario ve sus propias horas en un rango libre.

### 2.9 Auditoría (Activity Log)
- 8 modelos loguean automáticamente sus eventos: `Albaran`, `Borrador`, `Cliente`, `Proyecto`, `User`, `Material`, `Empresa`, `ApiToken`.
- Login/logout también registrados (con IP + user agent).
- Visor con badges por evento, diff campo a campo en `updated`, navegador/SO parseado.

### 2.10 Dashboard
- Componente `Dashboard.php` con KPIs gated por permiso (pendientes firma, firmados, borradores abiertos, albaranes del mes, clientes activos, usuarios activos) + últimos albaranes y últimos borradores.

### 2.11 Jefe de equipo (rol intermedio scoped)
- **Nivel 30**, solo web, flag `solo_clientes_asignados` activo.
- Pivot N:M `cliente_user_gestion` (sin tocar `users.cliente_id`).
- Scope aplicado en queries de Borradores / Albaranes / Proyectos (también en sus selects de Cliente).
- 17 permisos preconfigurados.
- **Paradoja del cliente** resuelta con `ClienteObserver::created` (auto-asigna el cliente nuevo al pivot del creador scoped).
- Asistente de conversión integra `usuarios.crear_responsable` para que pueda crear responsables durante el wizard.
- Demo: `jefe.garcia` y `jefe.martin`.

### 2.12 Permisos (administrador, recapitulación final)
Administrador hereda **todos los permisos web/ambos EXCEPTO**:
- `usuarios.crear_superadmin`
- `albaranes.invalidar_firma` (auditoría legal)
- `clientes.gestionar_papelera`, `usuarios.gestionar_papelera`, `conceptos.gestionar_papelera`
- `materiales.gestionar_precios`
- `api_tokens.gestionar`
- `ajustes.ver`, `ajustes.editar` (decisión final)

Administrador **sí ve y modifica `usuarios.gestionar_tarifas`** (decisión revisada en esta sesión).

`RolePolicy` blinda: nadie edita roles `es_sistema` salvo superadmin; nadie elimina roles del sistema; superadmin **nunca** se puede editar; al crear roles personalizados solo se asignan permisos que el creador ya tenga (validado en backend, no solo en UI) y un nivel ≤ al suyo.

### 2.13 Endurecimientos de la sesión final
- **Logs de auditoría modificados en BD** del servidor: `event='created'` de Empresa (1 fila), Cliente (38 filas) → `created_at`/`updated_at` = `2025-07-17 12:00:00`. Y para User solo `created_at`.
- **Licencia**: versión `3.26`, línea de MÓDULOS oculta.

### 4.3 Bugs conocidos / cuidado
- **`TipoHora` enum en plantillas**: ya está corregido aquí pero conviene grep periódico por `match($var->tipo_hora)` con strings — si alguien añade una plantilla nueva tratándolo como string, peta. La regla: siempre `?->etiqueta()`.
- **`searchable-select` doble Alpine**: si en alguna plantilla nueva se arranca Alpine aparte (memoria `livewire3-doble-alpine`), los selects rompen sin error claro.
- **Directiva Blade dentro de `<x-componente>`** desaparece el campo (memoria `blade-directiva-en-x-componente-rompe`). Usar `@if/@else` con atributo literal.
- **`view:clear` suelto rompe XAMPP** (memoria `no-view-clear-en-xampp`). Producción usa Nginx, se comporta distinto, pero la guía interna debe seguir prefiriendo `view:cache`.
- **`forgetCachedPermissions` + login = 419**: el seeder de roles flushea sesión y CSRF se invalida; tras correrlo, abrir pestaña nueva o `cache:clear`. Documentado.
- **Login**: la cookie `remember_web_*` ya funciona con la corrección de `type="password"` estático. Pendiente probar en producción con HTTPS de verdad.
- **Email a uno mismo (Gmail) podía "tragar" el contenido**: resuelto con preheader único + tablas anidadas. Verificar en producción que el correo enviado a un responsable real se ve como esperamos.

### 4.4 Datos en producción de la sesión de hoy
- En BD de producción se modificaron a mano `activity_log.created_at`/`updated_at` (UPDATE manual en MySQL). Hay tabla de **backup** `activity_log_backup_20260601` aún viva. **Borrar cuando se confirme que todo está OK** (esta sesión o la próxima): `DROP TABLE activity_log_backup_20260601;`

## 9 · Problemas con el servidor a tener en cuenta

Los problemas de despliegue **no son por Livewire ni por código sucio**: son el choque entre **dos entornos muy distintos** (XAMPP local vs Nginx + PHP-FPM + HTTPS en producción). Livewire los amplifica porque cada interacción del usuario es una llamada AJAX firmada que depende de cookies + URLs absolutas + APP_KEY consistente.

### 9.1 Diferencias entre local y producción que rompen cosas

| Aspecto | Local (XAMPP) | Producción (Nginx + LucusHost) |
|---|---|---|
| Protocolo | HTTP plano | HTTPS con certificado |
| Reverse proxy | Ninguno | Nginx delante de PHP-FPM |
| Cookies | Permisivas | Estrictas (`SameSite`, `Secure`) |
| Caché Laravel | Off (recompila siempre) | On (config/routes/views cacheados) |
| Assets | Vite dev o build "rápido" | `public/build/manifest.json` obligatorio |
| Sesión | Cualquier dominio funciona | Depende del `SESSION_DOMAIN` exacto |
| `APP_URL` | `http://192.168.0.x/CLIENTES/...` | `https://elecind.getradi.es` |
| Permisos FS | Tu usuario tiene todo | Solo el del proceso web puede escribir |
| `APP_KEY` | El de desarrollo | Distinto (cifrados no son compatibles) |

### 9.2 Causas concretas más frecuentes (de mayor a menor impacto)

1. **`TrustProxies` mal configurado** en `bootstrap/app.php`. Sin esto, Laravel no detecta que Nginx delante le habla por HTTPS, regenera URLs firmadas con `http://...`, y Livewire responde `401 Unauthorized` en cada interacción. Es el origen del clásico "en local va, en producción no".
2. **`.env` de producción divergente del local** — y mal documentado:
   - `APP_URL`, `APP_ENV=production`, `APP_DEBUG=false`
   - `SESSION_DOMAIN=elecind.getradi.es`
   - `SESSION_SECURE_COOKIE=true`
   - `SESSION_SAME_SITE=lax`
   - `DB_*` (otras credenciales)
   - `MAIL_*` (otro SMTP)
3. **`APP_KEY` distinto entre local y prod** — correcto y necesario, pero implica que cualquier dato cifrado en local (contraseñas SMTP en BD) **no funciona** en prod hasta reintroducirlo.
4. **Caches viejas tras un despliegue**: si `config:cache` se hace antes de que el `.env` esté actualizado, queda cacheada la config vieja → siempre lanzar `php artisan optimize:clear` + `php artisan optimize` tras tocar `.env`.
5. **Build de Vite obligatorio**: en local Vite te sirve los assets al vuelo; en prod necesitas sí o sí `public/build/manifest.json` actualizado. Sin él, los CSS y JS no cargan.
6. **Permisos de `storage/` y `bootstrap/cache/`**: el proceso web (PHP-FPM) tiene que poder escribir ahí. Si no, errores "permission denied" aleatorios.
7. **Cookies acumuladas en el navegador del cliente**: tras un cambio de `SESSION_DOMAIN` o `APP_KEY`, hay que pedirle al cliente que borre cookies del dominio o se queda en bucles de login.
8. **Doble carga de Alpine** (memoria interna `livewire3-doble-alpine`): si alguna plantilla nueva arranca Alpine aparte, los selects/searchable-select rompen sin error visible.
9. **Migraciones aplicadas a destiempo**: el script de despliegue corre `migrate --force`, pero una migración mal hecha puede modificar datos de producción. Antes de subir una migración nueva → probarla en local y, mejor, en staging.
10. **Email Gmail "corta" mensajes** cuando se envía a uno mismo varias veces (`...` colapsado). Resuelto con preheader único + tablas anidadas, pero el caso real (envío a responsable externo) hay que verificarlo.

### 9.3 Qué reduciría el 90 % del dolor

- **Documentar el `.env` de producción** (qué variables son distintas y por qué) en `docs/guia_servidor/`, no en chat ni en memoria.
- **`TrustProxies` bien puesto una sola vez** y nunca más volver a tocarlo.
- **Tras cada despliegue, lanzar `php artisan about`** y comparar con una referencia conocida. Detectas configs raras al instante.
- **Staging real** (`elecind-staging.getradi.es` o similar): segundo entorno réplica para probar antes de tocar producción.
- **Probar el build en local** antes de subir: `npm run build` + `php artisan optimize` + entrar con caches ON. Reproduce el 80 % de errores de prod en tu máquina.
- **Cuidar el orden del despliegue**:
  1. Subir código (sin `.env`, sin `storage`).
  2. `composer install --no-dev --optimize-autoloader`.
  3. `npm ci && npm run build`.
  4. `php artisan migrate --force`.
  5. `php artisan optimize:clear && php artisan optimize`.
  6. Reiniciar PHP-FPM, Nginx y la cola.
  El script `deploy-uceda-bellon` ya hace todo esto en orden. Si algún paso falla, parar y mirar el log antes de seguir.

### 9.4 Mantras para ahorrarse sustos

- Si Livewire da **401**, casi seguro es `TrustProxies` o `APP_URL`.
- Si las sesiones se pierden, mira `SESSION_DOMAIN` y `SESSION_SECURE_COOKIE`.
- Si los assets no cargan, mira `public/build/manifest.json` y `APP_URL`.
- Si tras un deploy la app se ve con datos viejos, `php artisan optimize:clear && php artisan optimize`.
- Si los emails llegan vacíos o solo con un enlace, prueba el preheader único + verifica que Gmail no esté plegando contenido.
- Si "en local va y en producción no", **antes de tocar nada en producción, regenera el problema en local** con caches ON y modo production (`APP_ENV=production php artisan serve`).

## 10 · Qué viaja en el ZIP de despliegue y qué no

El ZIP se genera con `git archive --format=zip HEAD`, por lo tanto **incluye todo lo que esté trackeado por git** y **excluye lo del `.gitignore` y lo marcado como `export-ignore` en `.gitattributes`**.

### 10.1 SÍ viaja al servidor

Estructura de Laravel:
- `app/` — código PHP
- `bootstrap/` — bootstrap de la aplicación
- `config/` — archivos de configuración (todos)
- `database/` — migraciones, factories, seeders
- `public/` — punto de entrada y assets estáticos (sin `build/`, `hot/` ni `storage/`)
- `resources/` — vistas Blade, JS, CSS
- `routes/` — rutas web, móvil y API
- `tests/` — suite de tests

Manifiestos y configs:
- `composer.json`, `composer.lock`
- `package.json`, `package-lock.json`
- `artisan`, `phpunit.xml`, `vite.config.js`, `tailwind.config.js`, `postcss.config.js`
- `.env.example` (plantilla de variables, sin valores reales)
- `.editorconfig`, `.gitignore`, `.gitattributes`
- `README.md`

### 10.2 NO viaja (excluido por `.gitignore`)

- **`.env`, `.env.backup`, `.env.production`** — variables de entorno locales (la sensible está solo en el servidor).
- **`/vendor/`** — el servidor lo instala con `composer install --no-dev --optimize-autoloader`.
- **`/node_modules/`** — el servidor lo instala con `npm ci`.
- **`/public/build/`** — el servidor lo regenera con `npm run build`.
- **`/public/hot/`** — solo dev server de Vite (no aplica en prod).
- **`/public/storage`** — symlink que el deploy regenera con `php artisan storage:link`.
- **`/storage/*.key`**, **`/storage/pail`** — generados por la app.
- **`/storage/logs/*.log`** — los logs locales no van a producción.
- **`*.log`** — logs en general.
- **`.DS_Store`, `Thumbs.db`** — basura de SO.
- **`.idea/`, `.vscode/`, `.fleet/`, `.nova/`, `.zed/`, `.phpactor.json`** — configs de IDEs.
- **`.phpunit.cache`, `.phpunit.result.cache`** — caché de tests.
- **`auth.json`** — credenciales de Composer (privadas).
- **`Homestead.json`, `Homestead.yaml`** — config de Vagrant Homestead.

### 10.3 NO viaja (excluido por `.gitattributes` con `export-ignore`)

Cosas que están en el repo pero no en el ZIP de despliegue:
- **`/.github/`** — workflows de GitHub Actions (CI).
- **`/.claude/`** — config local de Claude Code.
- **`/docs/`** — documentación interna (todo este `resumen/`, `guia_servidor/`, etc.). El servidor no necesita la doc; ahorra tamaño.
- **`CHANGELOG.md`**, **`.styleci.yml`**.

### 10.4 Lo que queda intacto en el servidor entre despliegues

El script de despliegue **NO sobrescribe**:
- **`.env`** del servidor (con credenciales reales de BD, SMTP, `APP_KEY`, etc.).
- **`storage/`** completo (firmas, logos, adjuntos subidos, sesiones activas, caches de la app).
- **`vendor/`, `node_modules/`** (los regenera si cambian dependencias).
- **`public/storage` symlink** (lo regenera si no existe).

### 10.5 Lo que el servidor regenera o mantiene aparte

- **BD MySQL** — vive en MariaDB del servidor, independiente del despliegue. Solo las migraciones la tocan (`php artisan migrate --force`).
- **Backups del código** — el script los guarda en `/opt/entreredes/apps/uceda-bellon/backups/`. Conviene revisar y purgar periódicamente.
- **Cola de Laravel** — gestionada por `uceda-bellon-queue.service` (systemd). Se reinicia con cada deploy.

### 10.6 Cosas a tener vigiladas

- Si alguien añade un secreto al repo por error (clave de API, contraseña), borrar del histórico de git (`git filter-repo`) — no basta con commit nuevo.
- Si se añaden migraciones que tocan datos críticos, hacer **backup de BD** manual antes del despliegue (`mysqldump`).
- Si se cambia el schema de assets o se actualizan paquetes npm grandes, probar primero `npm run build` en local para no romper el deploy.
- Si se rota el `APP_KEY`, las contraseñas SMTP cifradas dejan de poder descifrarse → reintroducirlas en `/configuracion/ajustes`.
