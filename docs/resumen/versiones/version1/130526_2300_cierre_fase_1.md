### 👥 CRUD `Usuarios` (web) — la parte más compleja
Pantalla `/usuarios` con 7 columnas y 4 filtros combinables (Estado · Tipo · Rol · Empresa cliente). Modal alta/edición en grid 2 columnas dividido en 2 secciones (Datos personales + Acceso y rol).

**Funcionalidades clave:**
- **Autosugerencia de username** al escribir el nombre: "Juan" → `juan`; si existe → `juan.2`, `juan.3`, etc. (incluye soft-deleted para evitar colisiones al restaurar). Solo se aplica en alta y si el usuario no ha tocado manualmente el campo.

- Externos requieren `empresa_cliente_id`; los internos no lo permiten.
- Password opcional en edición (mantiene la anterior si va vacío).
- Soft delete + restore.

**22 tests feature** cubriendo: listado, autosugerencia (3 variantes), duplicados (3 variantes), jerarquía, validaciones, externos, filtros, búsqueda.

- **Caché de request** en el helper `Branding` (evita N consultas por petición) + `Branding::limpiarCache()` invocado al guardar.

**13 tests feature** cubriendo: acceso, redirect trabajador, singleton, defaults, edición datos básicos, validación colores hex, validación token, upload de logo (3 escenarios), Branding helper.

### 🧹 Refactor arquitectónico: campo `acceso` → fuente única en el rol
Tras observar el cliente que el campo `acceso` del User era redundante (el rol ya lo tiene), se hizo un refactor limpio en la misma sesión:

**Antes:**
- `users.acceso` (web/movil/ambos) — duplicaba el `acceso` del rol.
- Middleware leía el del User.
- Permitía estados incoherentes: `trabajador` con `acceso=web`, etc.
- Bug en `EnsureMobileAccess`: comparaba con `'mobile'` (inglés) cuando el enum estaba en español → móvil nunca pasaba el check salvo si era `'ambos'`.

**Después:**
- Columna `users.acceso` **eliminada** de la migración.
- Helpers nuevos en `User`: `tieneAccesoWeb()` y `tieneAccesoMovil()` que consultan los roles.
- Middlewares `EnsureWebAccess` / `EnsureMobileAccess` refactorizados para usar esos helpers.
- **Bug `mobile`→`movil` corregido** de paso.
- Field "Acceso" eliminado del modal de Usuario; en la tabla ahora se muestra el acceso heredado del rol.
- **9 tests feature nuevos** para los middlewares (no había cobertura).

**Implicación de negocio**: la regla "solo superadmin puede tener acceso `ambos`" se cumple automáticamente porque sólo el rol `superadmin` tiene ese valor en la BD. No hay forma de crear un usuario con `acceso=ambos` salvo asignándole el rol `superadmin`.

## 🎨 Decisiones de diseño confirmadas en esta iteración

11. **Master-detail singleton** para configuración empresa: una pantalla `/configuracion/empresa` con `mount()` que invoca `ConfiguracionEmpresa::actual()` (firstOrCreate con defaults). Patrón replicable para futuros singletons.
12. **CSS vars inyectadas en `<head>` desde BD**: el layout web pinta `<style>:root { --c-primary-700: …; --c-accent-100: …; }</style>` leyendo del helper `Branding`. Cambiar colores en runtime sin rebuild. Caché de request para evitar N+1.
13. **Logo en disco `public` (no medialibrary)**: para un único activo singleton, `Storage::disk('public')->store('branding', ...)` es más simple y suficiente. Reservamos medialibrary para colecciones polimórficas (Fase 2: fotos de incidencias).
14. **Duplicados como avisos no bloqueantes**: en Usuarios, los duplicados email/DNI/CIF NO impiden guardar. Se muestra un modal con opciones "Usar este" (abre el existente) o "Crear nuevo igualmente". Patrón replicable para otros CRUDs donde puedan haber colisiones legítimas.
15. **Jerarquía por nivel** en Policy: `UserPolicy::nivelDe($user)` lee el `max('nivel')` de sus roles y se compara con el `nivel` del target. Scoping del listado vía `whereDoesntHave('roles', fn ($q) => $q->where('nivel', '>', $nivelActual))`.
16. **Autosugerencia con flag manual**: `usernameTocadoManual` se enciende cuando `updatedFormUsername` detecta que el valor difiere del sugerido. Sin Alpine en el cliente, todo server-side y testeable.
17. **Sidebar con children filtrados por permiso individual**: cada subitem se filtra por su propio `permission`; si el padre es un contenedor (`route=null`) y todos sus hijos están filtrados, también se oculta.
18. **Fuente única de `acceso`**: el rol manda. El usuario no tiene campo `acceso` propio. Cambiar el `acceso` de un rol reasigna automáticamente quién entra dónde, sin tocar usuarios uno por uno.
